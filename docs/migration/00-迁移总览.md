## 关键架构决策（迁移前需确认）

### 1. 离线优先 vs 在线优先
- **旧方案**: SQLite 本地数据库 + 自定义 Sync Engine（复杂度极高）
- **建议**: TanStack Query 缓存 + 关键数据离线支持（降低复杂度）
- **需确认**: 哪些功能必须离线可用？

### 2. 状态表设计
- **旧方案**: 4 个独立状态表 (project_statuses, defect_statuses, statuses, severities)
- **建议**: 合并为统一 `status_types` 表 + TypeScript 枚举约束
- **需确认**: 是否采用统一状态表？

### 3. 文件存储
- **旧方案**: Azure Blob Storage
- **建议**: S3 兼容服务（更灵活）或保持 Azure
- **需确认**: 选择哪个存储方案？

### 4. 推送通知
- **旧方案**: Azure Notification Hub
- **建议**: Expo Notifications (简化移动端集成) 或 Firebase Cloud Messaging
- **需确认**: 选择哪个推送方案？

### 5. 功能裁剪
旧项目包含以下功能，部分可能不需要立即迁移：
- [ ] 供应链管理 (supply-chains)
- [ ] 用户封锁 (user-blocks)
- [ ] 实时翻译 (realtime-translation)
- [ ] AI 语音识别 (voice-recognition)
- [ ] 演示模式 (mock-auth / demo mode)
- [ ] 社交关系 (connections / relationships)
- [ ] 报告生成 (reports)

## 旧项目问题分析（需优化）

### 后端问题
1. **28+ 模块过于分散** - 部分模块可合并 (如 defects + defect-updates)
2. **class-validator DTO 冗余** - 大量模板代码，迁移到 Zod 更简洁
3. **手动 JWT 管理** - token 黑名单、刷新逻辑复杂，Better Auth 已内置
4. **Azure 耦合过重** - Auth、Storage、Notifications 全绑 Azure
5. **缺少类型安全 API** - REST + Swagger 无法保证前后端类型一致

### 移动端问题
1. **17 个 Zustand store 互相依赖** - 状态管理过于碎片化
2. **自定义 Sync Engine 复杂度高** - 107KB 的 init.ts，维护困难
3. **52 个 service 文件** - 职责不清晰，部分重复
4. **组件目录过多 (42+)** - 缺乏统一组织规范
5. **lib/stores 与 /stores 目录重复** - 代码组织混乱
