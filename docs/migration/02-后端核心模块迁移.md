# 步骤 02：后端核心模块迁移（tRPC 路由）

## 概述

将旧项目的 REST Controller/Service 模式迁移为 NestJS + tRPC Router 模式。仅迁移核心模块。

## 迁移范围

### 核心模块（本步骤）
| 旧模块 | tRPC Router | 功能 |
|--------|-------------|------|
| `auth/` | 由 Better Auth 处理 | 认证登录 |
| `users/` | `user.router.ts` | 用户 CRUD + 个人信息 |
| `projects/` | `project.router.ts` | 项目 CRUD |
| `project-members/` | `project-member.router.ts` | 成员管理 |
| `tasks/` | `task.router.ts` | 任务 CRUD |
| `subtasks/` | `subtask.router.ts` | 子任务 CRUD |
| `defects/` | `defect.router.ts` | 缺陷管理 |
| `statuses/` + `project-statuses/` | `status-type.router.ts` | 统一状态查询 |

### 不迁移（Better Auth 已覆盖）
| 旧模块 | 原因 |
|--------|------|
| `auth/auth.controller.ts` | Better Auth 统一处理 login/logout/session |
| `auth/azure-auth.service.ts` | 废弃 Azure OAuth，使用 Better Auth 社交登录 |
| `auth/sms.service.ts` | Better Auth Phone Plugin 替代 |
| `auth/services/token-blacklist.service.ts` | Better Auth session revoke 替代 |
| `common/guards/jwt-auth.guard.ts` | 使用 Better Auth session 中间件 |

### 架构变更

#### REST → tRPC 对照

```
旧: @Controller('projects') + @Get() / @Post() / @Patch() / @Delete()
    + class-validator DTO + Swagger 装饰器
    
新: @Router({ alias: 'project' }) + @Query() / @Mutation()
    + Zod schema 验证 + 自动类型推导
```

#### 示例对比

**旧代码 (REST)**:
```typescript
@Controller('projects')
export class ProjectsController {
  @Get()
  @UseGuards(JwtAuthGuard)
  async findAll(@Query() dto: PaginationDto) {
    return this.projectsService.findAll(dto);
  }

  @Post()
  @UseGuards(JwtAuthGuard)
  async create(@Body() dto: CreateProjectDto) {
    return this.projectsService.create(dto);
  }
}
```

**新代码 (tRPC)**:
```typescript
@Router({ alias: 'project' })
export class ProjectRouter {
  @Query({ input: paginationSchema, output: projectListSchema })
  async list(@Input() input: PaginationInput) {
    return this.projectService.findAll(input);
  }

  @Mutation({ input: createProjectSchema, output: projectSchema })
  async create(@Input() input: CreateProjectInput) {
    return this.projectService.create(input);
  }
}
```

## 任务

将旧项目的 REST API 迁移为 tRPC Router。按以下顺序创建模块：

### Phase 1: user 模块
1. 创建 `src/modules/user/user.schema.ts`
   - 定义 Zod schema: getUserInput, updateUserInput, userOutput, userListOutput
   - 参考旧项目 users/dto/ 中的字段定义

2. 创建 `src/modules/user/user.service.ts`
   - 注入 DATABASE_CONNECTION (Drizzle)
   - 实现: findById, findByEmail, updateProfile, listUsers
   - 使用 Drizzle 查询语法，参考旧项目 users.service.ts 的业务逻辑
   - 不需要迁移: 旧项目的 email-verification (Better Auth 处理)

3. 创建 `src/modules/user/user.router.ts`
   - @Router({ alias: 'user' })
   - @Query() me - 获取当前用户 (从 Better Auth session)
   - @Query() getById - 按 ID 获取
   - @Mutation() updateProfile - 更新个人信息
   - @Query() list - 分页列表

4. 创建 `src/modules/user/user.module.ts`
   - 导入 DatabaseModule
   - 注册 UserService, UserRouter

### Phase 2: project 模块
1. 创建 Zod schema (参考旧项目 projects/dto/)
2. 创建 Service (参考旧项目 projects.service.ts 的核心 CRUD 逻辑)
   - 不迁移: sync-bundle, address-validation, site-diary-scheduler
3. 创建 Router: list, getById, create, update, delete (软删除)
4. 创建 Module

### Phase 3: task 模块 + subtask 模块
1. 创建 Zod schema
2. 创建 Service (参考旧项目 tasks.service.ts)
   - 不迁移: dependency-cycle-validator (后期)
3. 创建 Router: list (按项目过滤), getById, create, update, delete
4. subtask 类似，parentId 关联 task

### Phase 4: defect 模块
1. 参考旧项目 defects.service.ts
2. CRUD + 状态变更 (合并 defect-updates 到 defect 模块)

### Phase 5: status-type 模块
1. 只读查询 Router: listByType, listByCategory
2. Seed 初始数据的 migration

### Phase 6: project-member 模块
1. 参考旧项目 project-members.service.ts
2. Router: listByProject, addMember, updateRole, removeMember

### 最后: 注册所有模块到 app.module.ts

## 规范要求
- 每个 Router 方法都要有 input/output Zod schema
- Service 使用 @Injectable() + 构造函数注入 DATABASE_CONNECTION
- 所有查询排除 deletedAt IS NOT NULL 的记录
- 分页使用统一的 paginationSchema (page, limit, 默认 page=1, limit=20)
- 错误使用 TRPCError 抛出 (NOT_FOUND, FORBIDDEN, BAD_REQUEST 等)
- 参考旧项目的业务逻辑但简化实现，去除不必要的复杂度
