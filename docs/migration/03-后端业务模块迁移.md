# 步骤 03：后端业务模块迁移

## 概述

在核心模块完成后，迁移二线业务模块。每个模块独立，可按需跳过。

## 模块清单与迁移策略

### A 类：建议迁移
| 旧模块 | 新 Router | 优化点 |
|--------|-----------|--------|
| `task-updates/` | 合并到 `task.router.ts` | 作为 task 的子路由，不再独立模块 |
| `defect-updates/` | 合并到 `defect.router.ts` | 同上 |
| `timesheets/` | `timesheet.router.ts` | 简化，去掉 projectTimesheetSettings |
| `attendance-records/` | `attendance.router.ts` | 保持独立 |
| `site-diaries/` | `site-diary.router.ts` | 去掉 scheduler，简化签名逻辑 |
| `photos/` + `attachments/` | `media.router.ts` | 合并为统一媒体模块 |
| `notifications/` | `notification.router.ts` | 简化，去掉推送实现 |

### B 类：建议延后
| 旧模块 | 原因 |
|--------|------|
| `supply-chains/` | 非核心功能 |
| `connections/` + `relationships/` | 社交功能，优先级低 |
| `messages/` | 可用第三方消息服务 |
| `reports/` | 分析报告，后期需求 |
| `ai/` | AI 功能，独立迭代 |
| `sync/` | 取决于离线方案决策 |
| `push-notifications/` | 依赖推送服务选型 |

### C 类：不迁移
| 旧模块 | 原因 |
|--------|------|
| `user-blocks/` | 可后期简单实现 |
| `redis/` | Better Auth 有自己的 session 管理 |
| `monitoring/` (Sentry) | 基础设施，不属于业务模块 |
| `config/` (Joi 验证) | NestJS ConfigModule 已处理 |

```
## 任务：迁移业务模块

### 1. task-updates 合并到 task 模块
- 在 task.schema.ts 中添加 taskUpdateSchema (userId, statusId, comment, createdAt)
- 在 tasks schema 中添加 task_updates 表
- 在 task.service.ts 中添加 addUpdate, getUpdatesByTaskId 方法
- 在 task.router.ts 中添加:
  - @Query() getUpdates({ taskId }) - 获取任务更新历史
  - @Mutation() addUpdate({ taskId, statusId, comment }) - 添加更新

### 2. defect-updates 合并到 defect 模块
- 同上模式，合并到 defect 模块

### 3. timesheet 模块
- 创建 `src/database/timesheets.schema.ts`:
  - timesheets 表: id, uuid, projectId, userId, date, startTime, endTime, hours, description, statusId, deletedAt, createdAt, updatedAt
- 创建 `src/modules/timesheet/` (router + service + schema + module)
- Router: list (按项目/用户过滤), create, update, delete
- 不迁移 projectTimesheetSettings（简化）

### 4. attendance 模块
- 创建 `src/database/attendance.schema.ts`:
  - attendance_records 表: id, uuid, projectId, userId, checkInAt, checkOutAt, latitude, longitude, notes, deletedAt, createdAt, updatedAt
- 创建 `src/modules/attendance/` (router + service + schema + module)
- Router: checkIn, checkOut, listByProject, listByUser

### 5. site-diary 模块
- 创建 `src/database/site-diaries.schema.ts`:
  - site_diaries 表: id, uuid, projectId, createdById, date, weather, notes, deletedAt, createdAt, updatedAt
  - site_diary_visitors 表: id, diaryId, visitorName, company, signatureUrl, createdAt
- 创建 `src/modules/site-diary/` (router + service + schema + module)
- Router: list, getById, create, update, delete
- 不迁移: scheduled exports, signature hash 验证

### 6. media 模块（合并 photos + attachments）
- 创建 `src/database/media.schema.ts`:
  - media 表: id, uuid, entityType ('project'|'task'|'defect'|'diary'), entityId, type ('photo'|'document'|'attachment'), fileName, filePath, fileSize, mimeType, uploadedById, deletedAt, createdAt
- 创建 `src/modules/media/` (router + service + schema + module)
- Router: listByEntity, upload, delete
- 文件存储暂用本地文件系统，后期接入 S3/Azure

### 7. notification 模块（简化版）
- 创建 `src/database/notifications.schema.ts`:
  - notifications 表: id, uuid, userId, title, body, type, entityType, entityId, isRead, readAt, createdAt
- 创建 `src/modules/notification/` (router + service + schema + module)
- Router: listMine, markAsRead, markAllAsRead, getUnreadCount
- 不迁移: 推送发送逻辑（后期接入推送服务）

### 8. 注册所有新模块到 app.module.ts

## 规范要求
- 同步骤 02 的规范
- 合并模块时保持单一职责
- media 模块使用多态关联 (entityType + entityId)
- notification 只做数据存储，不做推送发送
```
