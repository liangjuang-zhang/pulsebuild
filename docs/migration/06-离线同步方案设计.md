# 步骤 06：离线同步方案设计

## 概述

旧 App 使用 SQLite + 自定义 Sync Engine 实现完全离线优先。这是旧项目最复杂的部分（107KB init.ts），需要决定新项目是否保留这种架构。

## 旧方案分析

### 架构
```
服务端 (PostgreSQL) ←→ Sync API ←→ 客户端 Sync Engine ←→ 本地 SQLite
```

### 组件
- `lib/database/init.ts` (107KB) - SQLite 建表、20+ 张表
- `lib/database/*-repository.ts` (18 个 Repository 类)
- `lib/sync/sync-engine.ts` - 同步协调器
- `lib/sync/sync-pull-service.ts` - 拉取远程变更
- `lib/sync/sync-transport.ts` - HTTP 传输层
- `lib/sync/remote-sync-applier.ts` - 应用远程变更
- `lib/sync/track-change.ts` - 本地变更追踪
- 后端 `sync/sync.controller.ts` - 同步 API (全量/增量)
- 后端 `sync/optimistic-concurrency.ts` - 冲突解决

### 问题
1. **复杂度极高**: 维护 20+ SQLite 表 + 对应的 PostgreSQL 表，双份 schema
2. **冲突解决简陋**: 仅用 updatedAt 时间戳做 last-write-wins
3. **Bug 风险**: 同步逻辑涉及大量边界情况
4. **初始同步慢**: 全量数据下载耗时
5. **占用空间大**: 本地 SQLite 数据量不受控

## 方案对比

### 方案 A：在线优先 + TanStack Query 缓存（推荐）

```
tRPC API ←→ TanStack Query (内存缓存 + 持久化)
```

**优点:**
- 零额外复杂度，TanStack Query 自带缓存
- 自动失效与重新请求
- 使用 `@tanstack/query-persist-client-experimental` 可持久化到 AsyncStorage
- 离线时显示缓存数据，联网后自动刷新

**缺点:**
- 离线无法创建/编辑数据
- 适合大部分场景但不适合深度离线

**实现:**
```typescript
import { createAsyncStoragePersister } from '@tanstack/query-async-storage-persister';
import AsyncStorage from '@react-native-async-storage/async-storage';

const persister = createAsyncStoragePersister({
  storage: AsyncStorage,
});

// Query Client with persistence
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 5 * 60 * 1000,     // 5 分钟
      gcTime: 24 * 60 * 60 * 1000,  // 24 小时
    },
  },
});
```

### 方案 B：轻量离线支持 (TanStack Query + Mutation Queue)

```
tRPC API ←→ TanStack Query + Offline Mutation Queue
```

**优点:**
- 支持离线时创建/编辑（进入队列）
- 联网后自动提交队列中的操作
- 比完整 Sync Engine 简单很多

**缺点:**
- 冲突解决需要额外设计
- 队列操作可能失败需要用户干预

**实现:**
```typescript
import { onlineManager } from '@tanstack/react-query';

// Mutation 自带 retry + pausing
const mutation = trpc.task.create.useMutation({
  networkMode: 'offlineFirst', // 离线时暂停，联网后自动重试
});
```

### 方案 C：完整离线同步（迁移旧方案）

```
tRPC API ←→ Sync Engine ←→ SQLite (Expo SQLite)
```

**优点:**
- 工地环境完全离线可用
- 数据完整性最高

**缺点:**
- 复杂度最高，维护成本大
- 需要迁移整个 Sync Engine + 18 个 Repository
- 双份 Schema 同步维护

## Claude Code 提示词（方案 B）

```
## 任务：实现轻量离线 Mutation Queue

在方案 A 基础上增加:

### 1. 创建 `src/lib/offline-mutation-handler.ts`
- 使用 TanStack Query 的 mutation cache 持久化
- 配置 mutation 的 onMutate 做乐观更新
- 配置 onError 做回滚
- 配置 onSettled 做缓存同步

### 2. 创建 `src/hooks/use-offline-mutation.ts`
- 封装 trpc mutation + 乐观更新模式
- 离线时在 UI 上标记 pending 状态
- 联网后自动重试

### 3. 示例：离线创建任务
const createTask = trpc.task.create.useMutation({
  networkMode: 'offlineFirst',
  onMutate: async (newTask) => {
    await utils.task.list.cancel();
    const prev = utils.task.list.getData();
    utils.task.list.setData(undefined, (old) => [...(old || []),
      { ...newTask, id: 'temp-' + Date.now(), _pending: true }
    ]);
    return { prev };
  },
  onError: (err, newTask, context) => {
    utils.task.list.setData(undefined, context.prev);
  },
  onSettled: () => {
    utils.task.list.invalidate();
  },
});
```
