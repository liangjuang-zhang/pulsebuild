# 消息通知模块迁移方案

> 基于旧项目 `pulsebuild-service` 的通知系统，迁移至当前 NestJS + tRPC + Drizzle ORM 架构。

---

## 1. 旧项目通知系统分析

### 1.1 模块概览

旧项目包含 **3 个通知相关模块**:

| 模块 | 职责 | 关键依赖 |
|------|------|----------|
| `notifications/` | 应用内通知（CRUD + 邮件触发） | SendGrid、Push |
| `push-notifications/` | 设备推送注册与发送 | Expo Push SDK |
| `messages/` | 联系表单消息存储 | 无外部依赖 |

### 1.2 旧数据库Schema

**notifications 表**:
```
id: serial PK
uuid: uuid UNIQUE (前端标识)
user_id: integer FK → users(id)
notification_type: varchar(50)  -- 'TASK_ASSIGNED' | 'TASK_READY' | ...
related_task_uuid: uuid FK → tasks(uuid)
related_project_id: integer FK → projects(id)
related_project_uuid: uuid FK → projects(uuid)
title: varchar(100)
message: text
is_read: boolean DEFAULT false
created_at: timestamp
updated_at: timestamp
```

**push_installations 表**:
```
installation_id: varchar PK (客户端生成)
user_id: integer FK → users(id)
platform: 'apns' | 'fcm'
push_token: varchar
is_active: boolean
registered_at: timestamp
```

### 1.3 通知类型枚举

```
TASK_ASSIGNED, TASK_READY, TASK_COMPLETED, TASK_CREATED,
SUBTASK_CREATED, STATUS_CHANGED, DEFECT_CREATED, DEFECT_UPDATED,
PROJECT_UPDATED, PROJECT_MEMBER_ADDED,
CONNECTION_REQUESTED, CONNECTION_ACCEPTED, CONNECTION_DECLINED,
SITE_DIARY_GENERATED
```

### 1.4 旧项目核心流程

```
创建通知 → 写入DB → 异步并发:
  ├── SendGrid 邮件（带项目详情、附件）
  └── Expo 推送（查找用户所有活跃设备 → 批量发送）
```

## 2. 架构设计决策

### 2.1 当前项目 vs 旧项目对比

| 维度 | 旧项目 | 当前项目 | 迁移策略 |
|------|--------|----------|----------|
| 主键 | `serial` + `uuid` 双主键 | `text` (Better Auth风格) | 使用 `text` 主键，`crypto.randomUUID()` |
| 用户引用 | `integer` FK | `text` FK → `user(id)` | 改为 `text` 类型 |
| API | REST Controller | tRPC Router | 转换为 `@Query` / `@Mutation` |
| 邮件 | 模块内 SendGrid Service | 已有全局 `EmailModule` | 复用现有 `EmailService` |
| 推送 | Expo Push SDK 模块 | 未实现 | 新建 `PushNotificationModule` |
| 验证 | class-validator DTO | Zod Schema | 转换为 Zod |
| 同步 | WatermelonDB tombstone | 已有 `SyncModule` | 注册通知表 handler |

### 2.2 模块拆分策略

```
apps/backend/src/
├── database/
│   └── notification.schema.ts       # 通知 + 推送设备 Drizzle 表
├── modules/system/
│   ├── notification/                 # 应用内通知
│   │   ├── notification.module.ts
│   │   ├── notification.service.ts
│   │   ├── notification.router.ts
│   │   └── notification.schema.ts   # Zod DTOs
│   └── push/                        # 推送通知
│       ├── push.module.ts
│       ├── push.service.ts
│       ├── push.router.ts
│       ├── push.schema.ts           # Zod DTOs
│       └── expo-push.service.ts     # Expo SDK 封装
```

**设计原则**:
- `notification` 模块负责通知 CRUD + 通知创建时协调邮件/推送
- `push` 模块负责设备注册/注销 + 推送发送（由 notification 调用）
- `EmailModule` 已存在，直接复用，无需迁移
- `messages`（联系表单）暂不迁移，属于独立业务，非核心通知

### 2.3 权限设计

```
system.notification.list           # 查看自己的通知
system.notification.read           # 标记已读
system.notification.delete         # 删除通知
system.notification.unread-count   # 获取未读数

system.push.register               # 注册推送设备
system.push.unregister             # 注销推送设备
```

> 通知模块通常不需要严格的管理权限，用户只能操作自己的通知。可通过 service 层 userId 过滤实现隔离，而非权限装饰器。

## 3. 数据库 Schema 设计

### 3.1 notification.schema.ts (Drizzle)

```typescript
// apps/backend/src/database/notification.schema.ts
import { relations } from 'drizzle-orm';
import {
  pgTable, text, timestamp, boolean, index, varchar,
  pgEnum,
} from 'drizzle-orm/pg-core';
import { user } from './auth.schema';

/** 通知类型枚举 */
export const notificationTypeEnum = pgEnum('notification_type', [
  'TASK_ASSIGNED',
  'TASK_READY',
  'TASK_COMPLETED',
  'TASK_CREATED',
  'SUBTASK_CREATED',
  'STATUS_CHANGED',
  'DEFECT_CREATED',
  'DEFECT_UPDATED',
  'PROJECT_UPDATED',
  'PROJECT_MEMBER_ADDED',
  'CONNECTION_REQUESTED',
  'CONNECTION_ACCEPTED',
  'CONNECTION_DECLINED',
  'SITE_DIARY_GENERATED',
]);

/** 推送平台枚举 */
export const pushPlatformEnum = pgEnum('push_platform', ['apns', 'fcm']);

/** 通知表 */
export const notification = pgTable(
  'notification',
  {
    id: text('id').primaryKey(),                    // crypto.randomUUID()
    userId: text('user_id').notNull()
      .references(() => user.id, { onDelete: 'cascade' }),
    type: notificationTypeEnum('type').notNull(),
    title: varchar('title', { length: 100 }).notNull(),
    message: text('message'),
    relatedEntityType: varchar('related_entity_type', { length: 50 }),  // 'task' | 'project' | 'defect'
    relatedEntityId: text('related_entity_id'),     // 关联实体 ID（通用化）
    isRead: boolean('is_read').default(false).notNull(),
    readAt: timestamp('read_at'),
    createdAt: timestamp('created_at').defaultNow().notNull(),
    // WatermelonDB sync 支持
    lastModified: timestamp('last_modified', { precision: 3 })
      .defaultNow()
      .$onUpdate(() => new Date())
      .notNull(),
    deletedAt: timestamp('deleted_at'),             // 软删除（sync tombstone）
  },
  (table) => [
    index('notification_user_id_idx').on(table.userId),
    index('notification_user_read_idx').on(table.userId, table.isRead),
    index('notification_created_at_idx').on(table.createdAt),
    index('notification_entity_idx').on(table.relatedEntityType, table.relatedEntityId),
  ],
);

/** 推送设备安装表 */
export const pushInstallation = pgTable(
  'push_installation',
  {
    id: text('id').primaryKey(),                    // crypto.randomUUID()
    installationId: varchar('installation_id', { length: 128 }).notNull(),  // 客户端生成
    userId: text('user_id').notNull()
      .references(() => user.id, { onDelete: 'cascade' }),
    platform: pushPlatformEnum('platform').notNull(),
    pushToken: varchar('push_token', { length: 500 }).notNull(),
    isActive: boolean('is_active').default(true).notNull(),
    registeredAt: timestamp('registered_at').defaultNow().notNull(),
    updatedAt: timestamp('updated_at')
      .defaultNow()
      .$onUpdate(() => new Date())
      .notNull(),
  },
  (table) => [
    index('push_installation_user_idx').on(table.userId),
    index('push_installation_active_idx').on(table.userId, table.isActive),
  ],
);

/** 关系定义 */
export const notificationRelations = relations(notification, ({ one }) => ({
  user: one(user, {
    fields: [notification.userId],
    references: [user.id],
  }),
}));

export const pushInstallationRelations = relations(pushInstallation, ({ one }) => ({
  user: one(user, {
    fields: [pushInstallation.userId],
    references: [user.id],
  }),
}));
```

**设计改进点**（vs 旧项目）:

| 改进 | 说明 |
|------|------|
| 统一 ID 类型 | `text` 主键，与 Better Auth user 一致 |
| 通用关联 | `relatedEntityType` + `relatedEntityId` 替代多个 FK，更灵活 |
| 软删除 | `deletedAt` 支持 WatermelonDB sync tombstone |
| 已读时间 | 增加 `readAt` 字段，供审计使用 |
| 枚举类型 | 使用 `pgEnum` 替代 varchar，数据库层面约束 |

### 3.2 旧 → 新字段映射

| 旧字段 | 新字段 | 变更 |
|--------|--------|------|
| `id` (serial) | `id` (text UUID) | 主键类型变更 |
| `uuid` | 移除 | `id` 直接用 UUID |
| `user_id` (integer) | `userId` (text) | FK 类型跟随 user 表 |
| `notification_type` (varchar) | `type` (pgEnum) | 枚举约束 |
| `related_task_uuid` | `relatedEntityId` | 通用化 |
| `related_project_id` | `relatedEntityId` | 通用化 |
| `related_project_uuid` | `relatedEntityId` | 通用化 |
| - | `relatedEntityType` | 新增实体类型区分 |
| - | `readAt` | 新增已读时间 |
| - | `deletedAt` | 新增软删除 |
| - | `lastModified` | 新增同步追踪 |

## 4. 模块实现规划

### 4.1 Phase 1: 通知核心 (notification 模块)

**notification.schema.ts (Zod DTOs)**:
```typescript
// 查询输入
NotificationQueryInput: { unreadOnly?: boolean, limit?, offset?, entityType?, entityId? }

// 创建输入（内部调用，不暴露给前端）
CreateNotificationInput: { userId, type, title, message?, relatedEntityType?, relatedEntityId? }

// 响应
NotificationResponse: { id, type, title, message, relatedEntityType, relatedEntityId, isRead, readAt, createdAt }
```

**notification.service.ts**:
```
- create(input): 创建通知 + 触发邮件和推送（异步）
- list(userId, query): 获取用户通知（支持未读过滤、分页）
- markAsRead(userId, notificationId): 标记已读 + 设置 readAt
- markAllAsRead(userId): 批量标记已读
- delete(userId, notificationId): 软删除
- getUnreadCount(userId): 未读计数
- notifyTaskAssigned(userId, taskId, title, message): 业务便捷方法
- notifyTaskCompleted(userId, taskId, title, message): 业务便捷方法
- notifyStatusChanged(userId, entityType, entityId, title, message): 业务便捷方法
```

**notification.router.ts**:
```
tRPC Router (alias: 'notification'):
- list (Query): 当前用户通知列表
- unreadCount (Query): 当前用户未读数
- markAsRead (Mutation): 标记单个已读
- markAllAsRead (Mutation): 标记全部已读
- delete (Mutation): 删除通知
```

> 注意: `create` 不暴露为 tRPC 路由，仅由其他 service 内部调用。

### 4.2 Phase 2: 推送通知 (push 模块)

**expo-push.service.ts**: 封装 Expo Server SDK
```
- sendNotifications(messages[]): 批量发送 + ticket/receipt 处理
- isValidPushToken(token): 验证令牌格式
```

**push.service.ts**: 设备管理 + 推送协调
```
- registerInstallation(userId, installationId, platform, pushToken): 注册/更新设备
- unregisterInstallation(userId, installationId): 停用设备
- sendToUser(userId, title, message, data?): 查找活跃设备 → 批量推送
```

**push.router.ts**:
```
tRPC Router (alias: 'push'):
- register (Mutation): 注册推送设备
- unregister (Mutation): 注销推送设备
```

### 4.3 Phase 3: WatermelonDB Sync 集成

在 `SyncModule` 中注册 `NotificationSyncHandler`:
```
- queryForPull(userId, lastPulledAt): 返回 created/updated/deleted 通知
- applyPushChanges(userId, changes): 处理客户端离线变更（标记已读等）
```

## 7. 关键注意事项

### 7.1 安全
- 通知查询必须过滤 `userId`，确保用户只能访问自己的通知
- 推送令牌属于敏感数据，不应在 API 响应中暴露完整值
- `EXPO_ACCESS_TOKEN` 和 `SENDGRID_API_KEY` 仅通过环境变量注入

### 7.2 性能
- 通知创建后邮件和推送应异步执行（`Promise.all` 不 await 阻塞主流程）
- 高频场景考虑后续引入 BullMQ 消息队列
- 未读计数查询需要 `(userId, isRead)` 复合索引

### 7.3 不迁移项
- **messages 模块**（联系表单）：独立业务，非通知核心，后续按需迁移
- **邮件模板详细内容**：当前 EmailService 已足够，模板内容随业务模块逐步补充
- **REST API 端点**：全部转为 tRPC，不保留 REST 兼容

### 7.4 向后兼容
- 移动端已有的 WatermelonDB 通知表需要通过 `SyncModule` 同步
- 通知类型枚举应保持与旧项目一致，确保移动端解析兼容
- `relatedEntityType` + `relatedEntityId` 需要移动端适配（旧项目用多个 FK 字段）
