### 各组件职责分工（重要！）

方案B采用 **WatermelonDB** 作为核心数据层，这会改变原有技术栈中 **tRPC** 和 **TanStack Query** 的角色。

#### 数据类型分类与处理方式

```
┌────────────────────────────────────────────────────────────────┐
│              数据类型分类                                        │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   类型1: 离线业务数据（任务、项目、缺陷...）                      │
│   ───────────────────────────────────────────                 │
│   存储：WatermelonDB (本地 SQLite)                             │
│   查询：withObservations (实时订阅)                             │
│   同步：synchronize() → tRPC sync endpoint                     │
│   ───────────────────────────────────────────                 │
│   ❌ TanStack Query 不参与（被替代）                            │
│                                                                │
│   类型2: 非离线数据（用户配置、系统设置、临时查询）                │
│   ───────────────────────────────────────────                 │
│   存储：不持久化 / AsyncStorage                                 │
│   查询：TanStack Query + tRPC                                  │
│   同步：不需要（每次都从服务端获取）                              │
│   ───────────────────────────────────────────                 │
│   ✅ TanStack Query + tRPC 仍然需要                            │
│                                                                │
│   类型3: 同步专用 endpoint                                      │
│   ───────────────────────────────────────────                 │
│   API：tRPC sync.pull / sync.push                              │
│   调用：synchronize() 内部使用                                  │
│   ───────────────────────────────────────────                 │
│   ✅ tRPC 提供 sync endpoint 类型安全                           │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

#### 各组件职责变化对比

| 组件 | 方案A 职责 | 方案B 职责 | 变化说明 |
|------|-----------|-----------|---------|
| **tRPC** | 所有 API endpoint 类型安全 | sync endpoint 类型安全 + 非离线 endpoint | ⚠️ 职责减少，常规 CRUD endpoint 被 sync endpoint 替代 |
| **TanStack Query** | 所有服务端状态管理 | 仅非离线数据查询 | ⚠️ 职责大幅减少，离线业务数据由 WatermelonDB 管理 |
| **Zustand** | 客户端状态 + UI 状态 | 仅 UI 状态（可选） | ⚠️ 职责减少，业务状态由 Model 管理 |
| **WatermelonDB** | 无 | 离线业务数据 + 同步 | ✅ 新增核心组件 |
| **expo-sqlite** | 本地存储（手动管理） | 无（WatermelonDB 底层） | ⚠️ 被封装，不直接使用 |
| **Repository** | 手动封装 SQLite 操作 | 无（Model 替代） | ❌ 被替代 |
| **SyncEngine** | 手动 pushChanges/pullChanges | 无（synchronize 替代） | ❌ 被替代 |

#### 查询方式对比

**场景1: 查询任务列表（离线业务数据）**

```typescript
// ─────────────────────────────────────────
// ❌ 方案A: TanStack Query + tRPC
// ─────────────────────────────────────────

// 从服务端获取数据，缓存到 TanStack Query
const { data: tasks } = trpc.task.list.useQuery({ projectId });

// 离线时无法使用！必须手动实现本地存储和状态追踪


// ─────────────────────────────────────────
// ✅ 方案B: WatermelonDB withObservations
// ─────────────────────────────────────────

// 从本地 SQLite 获取数据，实时订阅变化
const tasks = useObservation(
  database.collections.get<Task>('tasks')
    .query(Q.where('project_id', projectId))
);

// ✨ 离线也能用！无需服务端
// ✨ 数据变化时 UI 自动刷新（实时订阅）
// ✨ 无需 TanStack Query
```

**结论**：对于离线业务数据，WatermelonDB **替代** TanStack Query。

```typescript
// apps/backend/src/modules/sync/sync.router.ts

@defineRouter()
export class SyncRouter {
  // Pull endpoint
  @procedure.input(z.object({
    lastPulledAt: z.number(),
  })).output(z.object({
    changes: z.any(),
    timestamp: z.number(),
  }))
  async pull(input: { lastPulledAt: number }) {
    // ...
  }
  
  // Push endpoint
  @procedure.input(z.object({
    changes: z.any(),
  })).output(z.void())
  async push(input: { changes: PushChanges }) {
    // ...
  }
}

// ─────────────────────────────────────────
// WatermelonDB synchronize() 使用 tRPC
// ─────────────────────────────────────────

export async function syncDatabase() {
  await synchronize({
    database,
    
    pullChanges: async ({ lastPulledAt }) => {
      // ✨ 使用 tRPC 调用 sync.pull endpoint
      const response = await trpc.sync.pull.query({ lastPulledAt });
      return { changes: response.changes, timestamp: response.timestamp };
    },
    
    pushChanges: async ({ changes }) => {
      // ✨ 使用 tRPC 调用 sync.push endpoint
      await trpc.sync.push.mutate({ changes });
    },
  });
}
```

## 二、核心概念详解

### 2.1 什么是"实时订阅"（withObservations）？

**实时订阅** 是 WatermelonDB 提供的一种响应式数据查询机制，让 UI 组件能够自动响应数据库变化而无需手动刷新。

#### 传统方式 vs 实时订阅

```typescript
// ─────────────────────────────────────────────────────
// ❌ 传统方式：手动刷新（方案A）
// ─────────────────────────────────────────────────────

function TaskListScreen() {
  const [tasks, setTasks] = useState([]);
  
  // 1. 手动加载数据
  const loadTasks = async () => {
    const data = await taskRepository.getAll();
    setTasks(data);  // 手动 set()
  };
  
  // 2. 创建任务后，需要手动刷新
  const createTask = async () => {
    await taskRepository.create({ name: '新任务' });
    await loadTasks();  // ← 必须手动调用！
  };
  
  // 3. 更新任务后，需要手动刷新
  const updateTask = async (id, changes) => {
    await taskRepository.update(id, changes);
    await loadTasks();  // ← 必须手动调用！
  };
  
  useEffect(() => {
    loadTasks();
  }, []);
  
  return <FlatList data={tasks} ... />;
}

// 问题：
// 1. 每次数据变化都需要手动调用 loadTasks()
// 2. 容易忘记刷新，导致 UI 数据过期
// 3. 多处修改数据时，需要多处手动刷新


// ─────────────────────────────────────────────────────
// ✅ 实时订阅：自动刷新（方案B）
// ─────────────────────────────────────────────────────

function TaskListScreen() {
  // ✨ useObservation 自动订阅数据库变化
  const tasks = useObservation(
    database.collections.get<Task>('tasks')
      .query(Q.where('project_id', projectId))
  );
  
  // 创建任务（无需手动刷新！）
  const createTask = async () => {
    await database.collections.get('tasks').create(task => {
      task.name = '新任务';
      // ✨ 写入数据库后，useObservation 自动检测变化
      // ✨ UI 自动重新渲染，无需任何手动操作
    });
  };
  
  // 更新任务（无需手动刷新！）
  const updateTask = async (task, changes) => {
    await task.update(() => {
      task.name = changes.name;
      // ✨ 更新数据库后，useObservation 自动检测变化
      // ✨ UI 自动重新渲染
    });
  };
  
  // ✨ 无需 useEffect 手动加载！
  // ✨ 无需 useState + setTasks！
  
  return <FlatList data={tasks} ... />;
}

// 优点：
// 1. 数据变化 → UI 自动刷新，无需手动干预
// 2. 不会忘记刷新，UI 永远与数据库同步
// 3. 代码更简洁，无需 useState + useEffect + loadTasks()
```

#### 实时订阅的工作原理

```
┌────────────────────────────────────────────────────────────────┐
│              实时订阅（withObservations）工作原理                │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   ┌───────────────────────────────────────────────────────┐   │
│   │  1. 组件渲染时                                         │   │
│   │     useObservation(query)                             │   │
│   │     ↓                                                  │   │
│   │     执行查询：SELECT * FROM tasks WHERE project_id=?   │   │
│   │     ↓                                                  │   │
│   │     返回结果：[{id:1, name:'任务A'}, ...]              │   │
│   │     ↓                                                  │   │
│   │     注册观察者：监听 tasks 表的变化                     │   │
│   └───────────────────────────────────────────────────────┘   │
│                           ↓                                    │
│   ┌───────────────────────────────────────────────────────┐   │
│   │  2. 其他地方修改数据                                   │   │
│   │     task.update(() => { task.name = '新名称' })        │   │
│   │     ↓                                                  │   │
│   │     SQLite 执行：UPDATE tasks SET name='新名称'        │   │
│   │     ↓                                                  │   │
│   │     WatermelonDB 发出通知：tasks 表有变化！            │   │
│   └───────────────────────────────────────────────────────┘   │
│                           ↓                                    │
│   ┌───────────────────────────────────────────────────────┐   │
│   │  3. 观察者收到通知                                     │   │
│   │     ↓                                                  │   │
│   │     重新执行查询：SELECT * FROM tasks WHERE project_id=?│   │
│   │     ↓                                                  │   │
│   │     返回新结果：[{id:1, name:'新名称'}, ...]           │   │
│   │     ↓                                                  │   │
│   │     React 重新渲染组件                                 │   │
│   └───────────────────────────────────────────────────────┘   │
│                           ↓                                    │
│   ┌───────────────────────────────────────────────────────┐   │
│   │  4. UI 显示最新数据                                    │   │
│   │     用户看到：任务名称已变为"新名称"                    │   │
│   │     ✨ 全程无需手动刷新！                              │   │
│   └───────────────────────────────────────────────────────┘   │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

#### 类比理解

**实时订阅** 类似于其他技术：

| 技术 | 类似概念 | 说明 |
|------|----------|------|
| TanStack Query | `useQuery` 自动刷新 | 服务端数据变化 → 自动重新请求 |
| React | `useState` + `useEffect` | 状态变化 → 自动重新渲染 |
| RxJS | `Observable.subscribe()` | 数据流变化 → 自动收到通知 |
| Firebase | `onSnapshot()` | 数据库变化 → 自动回调 |
| MobX | `@observable` + `observer` | 数据变化 → 自动渲染 |

**区别**：
- TanStack Query：监听**服务端**数据变化（需要网络请求）
- WatermelonDB：监听**本地 SQLite** 数据变化（无需网络，离线可用）

### 2.2 WatermelonDB 的魔法字段：自动状态追踪

```typescript
// 当你定义一个 Model：
export class Task extends Model {
  @text('name') name!: string;
  @text('status') status!: string;
}

// WatermelonDB 自动添加这些字段：
// ┌────────────────────────────────────────────────────┐
// │  _id: string           -- 主键                      │
// │  _status: string       -- 同步状态                  │
// │  _changed: string      -- 修改的字段列表            │
// └────────────────────────────────────────────────────┘

// 当用户调用 task.update() 时：
await task.update(() => {
  task.name = '新名称';   // ✨ 自动：_status = 'updated'
  task.status = 'done';  // ✨ 自动：_changed = 'name,status'
});

// 当用户调用 database.create() 时：
await database.collections.get('tasks').create(task => {
  task.name = '新任务';  // ✨ 自动：_status = 'created'
});

// 当用户调用 task.markAsDeleted() 时：
await task.markAsDeleted();  // ✨ 自动：_status = 'deleted'
```

**对比方案 A**：

| 操作 | 方案 A (手动) | 方案 B (WatermelonDB) |
|------|--------------|----------------------|

```typescript
// 步骤 1: 同步函数（超简洁！）
// apps/mobile/src/lib/sync/watermelon-sync.ts

import { synchronize } from '@nozbe/watermelondb/sync';
import { trpc } from '../trpc';

export async function syncDatabase() {
  // ─────────────────────────────────────────
  // ✨ synchronize() 自动处理所有逻辑：
  // 1. 扫描所有表，找到 _status != 'synced' 的记录
  // 2. 按 created/updated/deleted 分类
  // 3. 批量推送到服务端
  // 4. 拉取服务端增量数据
  // 5. 自动合并到本地
  // 6. 标记 _status = 'synced'
  // ─────────────────────────────────────────
  
  await synchronize({
    database,
    
    // ─────────────────────────────────────────
    // Pull：从服务端获取增量数据
    // ─────────────────────────────────────────
    pullChanges: async ({ lastPulledAt }) => {
      // lastPulledAt：上次同步时间戳（WatermelonDB 自动传递）
      
      const response = await trpc.sync.pull.query({
        lastPulledAt,
      });
      
      // 返回格式必须符合 WatermelonDB 要求
      return {
        changes: response.changes,
        timestamp: response.timestamp,
      };
    },
    
    // ─────────────────────────────────────────
    // Push：发送本地修改到服务端
    // ─────────────────────────────────────────
    pushChanges: async ({ changes }) => {
      // ✨ WatermelonDB 自动打包所有变更：
      // changes = {
      //   tasks: {
      //     created: [{ _id, name, status, ... }],  -- _status='created' 的记录
      //     updated: [{ _id, name, ... }],          -- _status='updated' 的记录
      //     deleted: [_id1, _id2],                  -- _status='deleted' 的记录 ID
      //   },
      //   projects: {...},
      // }
      
      await trpc.sync.push.mutate({
        changes,
      });
    },
    
    // ─────────────────────────────────────────
    // ✨ 冲突解决策略（可选自定义）
    // ─────────────────────────────────────────
    conflictsResolver: ({ local, remote, columnName }) => {
      // 默认：服务端优先
      return remote;
      
      // 自定义示例：本地优先
      // if (columnName === 'name') {
      //   return local;  // 名称用本地
      // }
      // return remote;   // 其他字段用服务端
    },
    
    // 发送已同步的记录 ID（可选）
    sendCreatedAsUpdated: true,
    
    // 日志（可选）
    log: true,
    onDidPull: ({ changes }) => {
      console.log('Pull completed:', changes);
    },
    onDidPush: ({ changes }) => {
      console.log('Push completed:', changes);
    },
  });
  
  // ✨ 同步完成后：
  // - 所有 _status 自动变为 'synced'
  // - UI 自动刷新（withObservations 自动监听）
}

import NetInfo from '@react-native-community/netinfo';

class SyncScheduler {
  private syncInterval: NodeJS.Timeout | null = null;
  
  start() {
    // ─────────────────────────────────────────
    // 监听网络状态
    // ─────────────────────────────────────────
    NetInfo.addEventListener((state) => {
      if (state.isConnected) {
        // ✨ 网络恢复时立即同步
        syncDatabase().catch(console.error);
      }
    });
    
    // ─────────────────────────────────────────
    // 定时同步（每 30 秒）
    // ─────────────────────────────────────────
    this.syncInterval = setInterval(() => {
      NetInfo.fetch().then((state) => {
        if (state.isConnected) {
          syncDatabase().catch(console.error);
        }
      });
    }, 30000);
  }
  
  stop() {
    if (this.syncInterval) {
      clearInterval(this.syncInterval);
    }
  }
}

export const syncScheduler = new SyncScheduler();
```

```typescript
// WatermelonDB 需要专门的 sync endpoint
// apps/backend/src/modules/sync/sync.router.ts

import { defineRouter } from 'nestjs-trpc';
import { z } from 'zod';

@defineRouter()
export class SyncRouter {
  constructor(
    private taskService: TasksService,
    private projectService: ProjectsService,
    private subtaskService: SubtasksService,
  ) {}
  
  // ─────────────────────────────────────────
  // Pull：返回服务端增量数据
  // ─────────────────────────────────────────
  @procedure.input(z.object({
    lastPulledAt: z.number(),
  })).output(z.object({
    changes: z.any(),
    timestamp: z.number(),
  }))
  async pull(input: { lastPulledAt: number }) {
    const timestamp = Date.now();
    
    // 获取所有表的增量数据
    const tasks = await this.taskService.findUpdatedSince(input.lastPulledAt);
    const projects = await this.projectService.findUpdatedSince(input.lastPulledAt);
    const subtasks = await this.subtaskService.findUpdatedSince(input.lastPulledAt);
    
    // 获取已删除的记录（从 tombstones 表）
    const deletedTasks = await this.taskService.findDeletedSince(input.lastPulledAt);
    const deletedProjects = await this.projectService.findDeletedSince(input.lastPulledAt);
    
    // ─────────────────────────────────────────
    // WatermelonDB 要求的格式：
    // {
    //   tasks: { created: [], updated: [], deleted: [] },
    //   projects: { created: [], updated: [], deleted: [] },
    // }
    // ─────────────────────────────────────────
    
    const changes = {
      tasks: {
        created: tasks.filter(t => t.created_at > input.lastPulledAt),
        updated: tasks.filter(t => t.updated_at > input.lastPulledAt && t.created_at <= input.lastPulledAt),
        deleted: deletedTasks.map(t => t.id),
      },
      projects: {
        created: projects.filter(p => p.created_at > input.lastPulledAt),
        updated: projects.filter(p => p.updated_at > input.lastPulledAt && p.created_at <= input.lastPulledAt),
        deleted: deletedProjects.map(p => p.id),
      },
      subtasks: {
        created: subtasks.filter(s => s.created_at > input.lastPulledAt),
        updated: subtasks.filter(s => s.updated_at > input.lastPulledAt && s.created_at <= input.lastPulledAt),
        deleted: [],
      },
    };
    
    return { changes, timestamp };
  }
  
  // ─────────────────────────────────────────
  // Push：接收客户端变更
  // ─────────────────────────────────────────
  @procedure.input(z.object({
    changes: z.any(),
  })).output(z.void())
  async push(input: { changes: PushChanges }) {
    const { changes } = input;
    
    // ─────────────────────────────────────────
    // 处理 tasks 变更
    // ─────────────────────────────────────────
    if (changes.tasks) {
      // 创建
      for (const created of changes.tasks.created) {
        await this.taskService.create({
          id: created._id,  // 使用客户端 ID（可选）
          name: created.name,
          status: created.status,
          project_id: created.project_id,
          created_at: created.created_at,
          updated_at: created.updated_at,
        });
      }
      
      // 更新
      for (const updated of changes.tasks.updated) {
        await this.taskService.update(updated._id, {
          name: updated.name,
          status: updated.status,
          updated_at: updated.updated_at,
        });
      }
      
      // 删除
      for (const deletedId of changes.tasks.deleted) {
        await this.taskService.delete(deletedId);
      }
    }
    
    // 处理其他表...
    if (changes.projects) {
      // 类似逻辑
    }
  }
}

export class TasksService {
  async findUpdatedSince(since: number): Promise<Task[]> {
    return this.db.query.tasks.findMany({
      where: (t, { gt }) => gt(t.updated_at, since),
    });
  }
  
  async findDeletedSince(since: number): Promise<{ id: string }[]> {
    // 从 tombstones 表获取已删除的记录
    return this.db.query.sync_tombstones.findMany({
      where: (t, { eq, gt }) => 
        eq(t.entity_type, 'task') && 
        gt(t.deleted_at, since),
      columns: { server_id: true },
    });
  }
```

### 表删除策略：markAsDeleted + Tombstones

```typescript
// ─────────────────────────────────────────────────────
// WatermelonDB 删除策略：软删除（markAsDeleted）
// ─────────────────────────────────────────────────────

// apps/mobile/src/screens/TaskList.tsx

const deleteTask = async (task: Task) => {
  // ✨ markAsDeleted() 自动：
  // 1. 设置 _status = 'deleted'
  // 2. synchronize() 时会推送到服务端
  // 3. 服务端确认后，本地记录才真正删除
  await task.markAsDeleted();
  
  // UI 自动刷新（withObservations 自动过滤 deleted 记录）
};

// ─────────────────────────────────────────────────────
// synchronize() 处理删除
// ─────────────────────────────────────────────────────

// WatermelonDB synchronize() 自动：
// 1. 扫描所有 _status = 'deleted' 的记录
// 2. 打包到 changes.tasks.deleted = [_id1, _id2]
// 3. 推送到服务端
// 4. 服务端删除后，本地记录自动清除

// ─────────────────────────────────────────────────────
// 服务端处理删除
// ─────────────────────────────────────────────────────

// apps/backend/src/modules/sync/sync.router.ts

async push(input: { changes: PushChanges }) {
  // 处理删除
  for (const deletedId of input.changes.tasks.deleted) {
    // 1. 删除服务端数据
    await this.taskService.delete(deletedId);
    
    // 2. 写入 tombstones 表（防止重复拉取）
    await this.syncService.recordTombstone('task', deletedId);
  }
}

// ─────────────────────────────────────────────────────
// Pull 时过滤 tombstones
// ─────────────────────────────────────────────────────

// apps/backend/src/modules/sync/sync.router.ts

async pull(input: { lastPulledAt: number }) {
  // 获取已删除的记录 ID
  const deletedTasks = await this.syncService.getTombstonesSince(
    'task', 
    input.lastPulledAt
  );
  
  const changes = {
    tasks: {
      created: [...],
      updated: [...],
      deleted: deletedTasks,  // ← 返回已删除的 ID
    },
  };
  
  return { changes, timestamp };
}

// ─────────────────────────────────────────────────────
// WatermelonDB Pull 时自动处理删除
// ─────────────────────────────────────────────────────

// synchronize() 拉取服务端数据时：
// 1. 收到 changes.tasks.deleted = [_id1, _id2]
// 2. 自动删除本地记录
// 3. UI 自动刷新
```

## 五、数据冲突处理

### WatermelonDB 内置冲突检测

```typescript
// ─────────────────────────────────────────────────────
// WatermelonDB synchronize() 内置冲突检测
// ─────────────────────────────────────────────────────

await synchronize({
  database,
  pullChanges: async ({ lastPulledAt }) => { ... },
  pushChanges: async ({ changes }) => { ... },
  
  // ─────────────────────────────────────────
  // ✨ 冲突解决策略（可选自定义）
  // ─────────────────────────────────────────
  conflictsResolver: ({ local, remote, columnName }) => {
    // 参数：
    // - local: 本地字段值
    // - remote: 服务端字段值
    // - columnName: 字段名
    
    // 默认策略：服务端优先
    return remote;
    
    // ─────────────────────────────────────────
    // 自定义策略示例：
    // ─────────────────────────────────────────
    
    // 1. 字段级别策略
    if (columnName === 'name') {
      return local;  // 名称保留本地
    }
    return remote;   // 其他字段用服务端
    
    // 2. 时间戳策略
    if (local.updated_at > remote.updated_at) {
      return local;  // 本地更新时间晚，用本地
    }
    return remote;
    
    // 3. 版本号策略
    if (local.version > remote.version) {
      return local;
    }
    return remote;
  },
  
  // ─────────────────────────────────────────
  // 冲突记录（可选）
  // ─────────────────────────────────────────
  onDidMigrate: ({ changes }) => {
    // 记录冲突处理结果
    console.log('Migrated:', changes);
  },
});

// ─────────────────────────────────────────────────────
// 冲突场景处理示例
// ─────────────────────────────────────────────────────

// 场景：
// - 用户 A 离线修改任务名称为 "已完成墙面"
// - 用户 B 在服务端修改同一任务名称为 "需要返工"
// - 用户 A 网络恢复，尝试同步

// WatermelonDB 自动处理：
// 1. 推送用户 A 的修改到服务端
// 2. 服务端检测到冲突（任务已被用户 B 修改）
// 3. 服务端返回最新数据（用户 B 的修改）
// 4. synchronize() 调用 conflictsResolver
// 5. 根据策略决定最终值（默认：服务端优先）
// 6. 更新本地数据，标记 _status = 'synced'
// 7. UI 自动刷新

// ─────────────────────────────────────────────────────
// 服务端冲突检测（可选）
// ─────────────────────────────────────────────────────

// apps/backend/src/modules/sync/sync.router.ts

async push(input: { changes: PushChanges }) {
  for (const updated of input.changes.tasks.updated) {
    const serverTask = await this.taskService.findById(updated._id);
    
    // 检测冲突：服务端更新时间晚于客户端
    if (serverTask.updated_at > updated.updated_at) {
      // 冲突！返回最新数据给客户端
      // synchronize() 会调用 conflictsResolver 处理
      // 这里不做处理，让客户端决定
    }
    
    await this.taskService.update(updated._id, {
      name: updated.name,
      status: updated.status,
      updated_at: Date.now(),
    });
  }
}
```

## 六、总结：方案 B 特点

### 优点

1. **自动化程度高**：_status、_changed 自动追踪，无需手动维护
2. **代码简洁**：Model 定义 + synchronize()，无需 Repository + SyncEngine
3. **UI 自动刷新**：withObservations 自动订阅，无需手动 set()
4. **冲突内置**：synchronize() 内置冲突检测和解决
5. **Migration 管理**：Schema 版本号 + 自动执行

### 缺点

1. **迁移成本高**：源项目 Repository 需重写为 Model
2. **后端改造**：需要实现专门的 sync endpoint（pull/push）
3. **学习成本**：WatermelonDB 概念需要学习
4. **调试难度**：抽象层多，问题定位难
5. **灵活性受限**：冲突策略、同步逻辑由 WatermelonDB 控制

### 适用场景

- 新项目，无历史代码
- 团队愿意学习新技术
- 需要快速实现离线功能
- 标准化的同步逻辑（无特殊需求）
