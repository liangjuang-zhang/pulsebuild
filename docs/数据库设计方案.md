# PulseBuild 后端数据库设计方案

> 本文档为 pulsebuild 新后端数据库设计分析，对比旧项目 pulsebuild-service 的 schema，提出优化建议。

---

## 1. User 表扩展分析

### 1.1 当前项目 User 表（Better Auth 核心）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | text | 主键（Better Auth 生成的字符串 ID） |
| name | text | 用户名 |
| email | text | 邮箱，唯一 |
| emailVerified | boolean | 邮箱是否验证 |
| image | text | 头像 URL |
| createdAt | timestamp | 创建时间 |
| updatedAt | timestamp | 更新时间（自动更新） |

### 1.2 pulsebuild-service User 表（旧项目）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | serial | 自增主键 |
| uuid | uuid | 唯一标识（分布式友好） |
| name | varchar(100) | 用户名 |
| email | varchar(100) | 邮箱 |
| pendingEmail | varchar(100) | 待验证邮箱（邮箱变更流程） |
| countryCode | varchar(8) | 国家区号（国际化） |
| phone | varchar(20) | 手机号 |
| jobTitle | varchar(120) | 职位 |
| companyName | varchar(120) | 公司名 |
| deletedAt | timestamp | 软删除标记 |
| createdAt | timestamp | 创建时间 |
| updatedAt | timestamp | 更新时间 |
| timezone | varchar(64) | 时区 |
| avatarUrl | text | 头像 URL |
| profilePhotoUrl | text | 详细头像 URL |
| emailVerifiedAt | timestamp | 邮箱验证时间 |
| emailVerificationTokenHash | varchar(64) | 验证 token hash |
| emailVerificationExpiresAt | timestamp | 验证过期时间 |
| onboardingCompletedAt | timestamp | 完成引导时间 |

### 1.3 建议扩展字段

| 字段名 | 类型 | 优先级 | 说明 |
|--------|------|--------|------|
| uuid | uuid | **高** | Better Auth 使用 text ID，建议增加 uuid 作为业务标识，兼容分布式场景 |
| phone | varchar(20) | **高** | 手机号，多渠道登录/找回密码 |
| countryCode | varchar(8) | **高** | 国家区号，国际化支持 |
| timezone | varchar(64) | **高** | 用户时区，影响任务调度、通知时间 |
| deletedAt | timestamp | **高** | 软删除标记，数据恢复需求 |
| jobTitle | varchar(120) | **中** | 职位，企业用户场景 |
| companyName | varchar(120) | **中** | 公司名，企业用户场景 |
| lastLoginAt | timestamp | **中** | 最后登录时间，安全审计 |
| status | varchar(20) | **中** | 用户状态（active/suspended/pending） |
| onboardingCompletedAt | timestamp | **低** | 完成引导时间，新用户流程 |
| locale | varchar(10) | **低** | 语言偏好 |

### 1.4 字段合并建议

- `avatarUrl` + `profilePhotoUrl` → 合并为 `image`（Better Auth 已有）
- `emailVerified` (boolean) → 可改为 `emailVerifiedAt` (timestamp)，记录验证时间

## 2. 表结构优化建议

### 2.1 可精简/合并的表

#### 状态表合并

旧项目有 4 个相似的状态表：

| 原表名 | 说明 |
|--------|------|
| project_statuses | 项目状态 |
| defect_statuses | 缺陷状态 |
| defect_severities | 缺陷严重程度 |
| statuses | 任务状态 |
| severities | 任务严重程度 |

**优化方案**：合并为通用状态表

```typescript
// 统一状态表
export const statusTypes = pgTable('status_types', {
  id: serial('id').primaryKey(),
  type: varchar('type', { length: 50 }).notNull(), // 'project', 'task', 'defect'
  category: varchar('category', { length: 50 }), // 'status', 'severity'
  name: varchar('name', { length: 50 }).notNull(),
  value: varchar('value', { length: 50 }).notNull(), // 枚举值
  orderIndex: integer('order_index'),
  color: varchar('color', { length: 20 }), // UI 显示颜色
  description: text('description'),
  createdAt: timestamp('created_at').defaultNow(),
}, (table) => [
  uniqueIndex('status_types_unique').on(table.type, table.category, table.value),
]);
```

**收益**：
- 减少 5 个表 → 1 个表
- 统一状态管理，便于扩展
- 减少 JOIN 查询复杂度

#### 媒体附件合并

旧项目已有 `mediaAttachments` 统一表，但仍有冗余表：

| 原表名 | 说明 |
|--------|------|
| photos | 任务照片（旧） |
| defect_media_attachments | 缺陷媒体 |
| task_update_media_attachments | 任务更新媒体 |

**优化方案**：统一使用 `mediaAttachments`

```typescript
// 统一媒体附件表（已有设计，直接使用）
export const mediaAttachments = pgTable('media_attachments', {
  id: serial('id').primaryKey(),
  uuid: uuid('uuid').defaultRandom().unique(),
  entityType: varchar('entity_type', { length: 50 }).notNull(), // 'task', 'defect', 'task_update'
  entityId: uuid('entity_id').notNull(),
  contextType: varchar('context_type', { length: 50 }), // 'update', 'defect'
  contextId: uuid('context_id'),
  fileName: varchar('file_name', { length: 255 }).notNull(),
  fileSize: integer('file_size').notNull(),
  mimeType: varchar('mime_type', { length: 100 }).notNull(),
  url: text('url').notNull(),
  thumbnailUrl: text('thumbnail_url'),
  width: integer('width'),
  height: integer('height'),
  duration: integer('duration'), // 视频时长
  caption: text('caption'),
  uploadedBy: integer('uploaded_by').notNull(),
  uploadedAt: timestamp('uploaded_at').defaultNow(),
}, (table) => [
  index('media_attachments_entity_idx').on(table.entityType, table.entityId),
]);
```

**收益**：
- 减少冗余表
- 统一文件管理逻辑
- 便于添加新的实体类型

#### 更新记录合并

旧项目已有 `updates` 统一表：

| 原表名 | 说明 |
|--------|------|
| task_updates | 任务更新 |
| defect_updates | 缺陷更新 |

**优化方案**：统一使用 `updates` 表

```typescript
// 统一更新表（已有设计）
export const updates = pgTable('updates', {
  id: uuid('id').defaultRandom().primaryKey(),
  updateType: varchar('update_type', { length: 20 }).notNull(), // 'task', 'defect'
  projectId: integer('project_id').notNull(),
  taskId: uuid('task_id'),
  subtaskId: uuid('subtask_id'),
  defectId: uuid('defect_id'),
  userId: integer('user_id'),
  userUuid: uuid('user_uuid'),
  userName: varchar('user_name', { length: 100 }),
  comment: text('comment'),
  oldStatusId: integer('old_status_id'),
  newStatusId: integer('new_status_id'),
  createdAt: timestamp('created_at').defaultNow(),
}, (table) => [
  check('updates_type_check', sql`update_type IN ('task', 'defect')`),
]);
```

### 2.2 可去除的表

| 表名 | 说明 | 去除原因 |
|------|------|----------|
| `_prisma_migrations` | Prisma 迁移记录 | 仅迁移工具使用，生产环境可隐藏或移除 |
| `user_types` | 用户类型 | 类型较少，可用 enum 或在 user 表增加 role 字段 |
| `task_types` | 任务类型 | 可合并到通用 type 表或用 enum |
| `project_types` | 项目类型 | 同上 |
| `relationships` | 用户关系 | 功能较少使用，可简化为 `connections` 表 |
| `login_codes` | 手机登录验证码 | 可用 Better Auth 的 `verification` 表替代 |
| `blacklisted_tokens` | JWT 黑名单 | Better Auth 已有 session 管理，可复用 |

### 2.3 需要保留的核心表

| 表名 | 说明 | 保留原因 |
|------|------|----------|
| `user` | 用户表（Better Auth） | 核心认证表 |
| `session` | 会话表（Better Auth） | 认证会话管理 |
| `account` | 账户表（Better Auth） | OAuth/密码登录 |
| `verification` | 验证表（Better Auth） | 验证码管理 |
| `projects` | 项目表 | 核心业务实体 |
| `tasks` | 任务表 | 核心业务实体 |
| `subtasks` | 子任务表 | 核心业务实体 |
| `defects` | 缺陷表 | 核心业务实体 |
| `project_members` | 项目成员 | 权限管理 |
| `connections` | 用户连接 | 社交功能 |
| `timesheets` | 工时表 | 考勤功能 |
| `site_diaries` | 施工日志 | 施工管理 |
| `notifications` | 通知表 | 用户通知 |

## 3. 新项目 Schema 设计建议

### 3.1 Better Auth 表保留

```typescript
// apps/backend/src/database/auth.schema.ts - 保持不变
export const user = pgTable('user', {
  id: text('id').primaryKey(),
  name: text('name').notNull(),
  email: text('email').notNull().unique(),

  image: text('image'),
  createdAt: timestamp('created_at').defaultNow(),

### 3.2 扩展 User Profile 表

  companyName: varchar('company_name', { length: 120 }),
  
  // 状态管理
  status: varchar('status', { length: 20 }).default('active'),
  deletedAt: timestamp('deleted_at'),
  lastLoginAt: timestamp('last_login_at'),
  
  // 用户引导
  onboardingCompletedAt: timestamp('onboarding_completed_at'),

  uniqueIndex('user_profile_phone_unique').on(table.countryCode, table.phone),
  index('user_profile_user_id_idx').on(table.userId),
  index('user_profile_uuid_idx').on(table.uuid),
]);
```

## 4. 迁移策略

### 4.1 阶段一：保留 Better Auth 表

- 不修改 Better Auth 核心表（user, session, account, verification）
- 新建 `user_profile` 表作为扩展

### 4.2 阶段二：业务表迁移

- 创建统一状态表 `status_types`
- 创建统一媒体表 `media_attachments`
- 迁移业务表（projects, tasks, subtasks, defects 等）

### 4.3 阶段三：数据迁移

- 从旧项目导出数据
- 转换 ID 格式（serial → uuid/text）
- 批量导入新数据库

## 5. 最佳实践

### 5.1 命名规范

- 表名：小写蛇形 `user_profile`
- 字段名：小写蛇形 `created_at`
- 外键：`{table}_id` 格式
- 索引：`{table}_{columns}_idx` 格式

### 5.2 审计字段

所有业务表统一添加：

```typescript
createdAt: timestamp('created_at').defaultNow(),
updatedAt: timestamp('updated_at').$onUpdate(() => new Date()),
createdBy: text('created_by'), // Better Auth user.id
deletedAt: timestamp('deleted_at'), // 软删除
```

### 5.3 软删除

- 使用 `deletedAt` timestamp
- 查询时默认过滤 `deletedAt IS NULL`
- 提供 `includeDeleted` 选项用于管理员查看

### 5.4 索引策略

- 外键字段必须有索引
- 查询频繁的组合字段建立复合索引
- 状态字段建立索引（filter 查询）
- 时间字段建立索引（排序查询）

## 6. 总结

| 类别 | 原表数 | 优化后 | 收益 |
|------|--------|--------|------|
| 状态表 | 5 | 1 | 减少 80% |
| 媒体表 | 3 | 1 | 减少 67% |
| 更新表 | 2 | 1 | 减少 50% |
| 迁移表 | 1 | 0 | 完全移除 |
| 登录验证 | 1 | 0 | 使用 Better Auth |

**总体优化**：
- Better Auth 核心表保持不变，确保认证功能稳定
- 业务表统一设计，减少冗余
- 扩展字段集中在 `user_profile` 表，保持 Better Auth 兼容性
- 统一状态/媒体/更新管理，提升可维护性
